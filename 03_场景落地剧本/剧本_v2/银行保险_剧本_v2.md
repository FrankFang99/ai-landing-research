## 三、5A 拆解（v2：每段都明确"5A 溯源" + "v1 升级点"）

### 1. Agile（敏捷） —— 4 周出 POC，6 周进灰度

**5A 溯源**：Agile = "从经营核心场景切入，快速构建数据指标体系"（苏春园 2018 CIO 时代学院大会原话）。"4 周出 POC、6 周进灰度"是把"快速构建"具体到银行业可执行的天数。

**核心动作**：把"做大模型平台"这种大命题切成 5 个 Sprint，每个 Sprint 1 个可演示的版本。

| Sprint | 周次 | 交付物 | 演示场景 | 5A 维度自检 |
| --- | --- | --- | --- | --- |
| Sprint 0 | Week 1 | 1 个场景的最小可用版本（FAQ 自动回复） | 行员在内部 IM 输入"信用卡账单分期"，AI 秒回标准话术 + 转人工按钮 | 1 周内出 POC ✅ Agile ✅ |
| Sprint 1 | Week 2-3 | 3 个场景接入（FAQ + 智能工单分类 + 坐席实时辅助） | 客服坐席通话时屏幕弹提示卡 + 推荐话术 + 合规预警 | 现场实机演示 ✅ |
| Sprint 2 | Week 4-5 | 5 个场景 + 灰度 20% 坐席 + badcase 看板 | 20% 坐席双轨运行（AI 辅助 + 人工主管备份），每日 badcase 自动归集 | 5 周扩场景 ✅ |
| Sprint 3 | Week 6 | 全量灰度 + 第一个自适应版本（基于 badcase 微调提示词） | 80% 坐席使用，AI 自动给主管发"今日典型 badcase 5 条"报告 | 6 周进工作流 ✅ |

**v1 升级点**：v1 只到"周级别 Sprint 划分"，v2 在第三节"6 周路线图"里给到**每日级 check-in 节奏**（42 个工作日，每个工作日有明确动作）。

**工具栈 v2 选型**（v1 用 LangChain + LangSmith，v2 新增）：

| 工具 | v2 推荐 | 适用环节 | 关键理由 |
| --- | --- | --- | --- |
| **底座大模型** | DeepSeek-V3 / Qwen3-235B / 智谱 GLM-4.6 / 商汤日日新 6.5 | Sprint 0-3 通用底座 | 国内合规 + 私有化部署 + 价格 1/10 GPT-4o |
| **Agent 编排** | **Dify v1.0**（v2 推荐升级）/ **Coze 扣子企业版** | Sprint 0-1 场景搭建 | Dify 1.0 支持多 Agent + 工作流编排，扣子直接对接钉钉/飞书 |
| **RAG 框架** | **LangChain 1.0**（2025-10 发布，引入 LCEL 2.0） / LlamaIndex 0.12 | Sprint 0 RAG 检索 | LangChain 1.0 用 LangGraph 处理复杂工作流，比 0.x 性能提升 30% |
| **多 Agent 协作** | **AutoGen 0.4**（微软，2025 重写） / **CrewAI 0.80** | Sprint 2 灰度扩场景 | AutoGen 0.4 重写为 Actor Model；CrewAI 强调"角色-任务-工具"三件套 |
| **提示词版本管理** | LangSmith（LangChain 生态）/ PromptLayer 2.0 | Sprint 0-3 全程 | 行内 GitLab 同步管理（提示词与代码同库） |
| **向量库** | Milvus 2.5（v1 沿用） / Qdrant 1.13 | Sprint 0-3 知识库 | 金融场景千万级向量，Milvus 2.5 性能提升 50% |
| **国产合规栈** | 阿里百炼平台 / 华为云盘古大模型 / 火山方舟 | Sprint 0 备选底座 | 国资委"信创目录"硬约束，金融客户必须有 2+ 国产底座备选 |

**关键纪律**：
- **每个 Sprint 结束必须现场演示**（实机跑，不是 PPT；客户在场签字）
- **Sprint 边界不允许滑动**（做不完就砍场景不做完，保证节奏）
- **演示前必须过合规审查**（金融场景一旦出合规事故整个项目停摆）

### 2. Apply（场景化） —— 拒绝"通用大模型套一切"

**5A 溯源**：Apply（应用版）↔ Accurate（原始版）= "针对'人货场'关键场景的多维指标体系"（苏春园原话）。银行场景的"人货场"对应"客户 × 产品 × 渠道"，必须 3 个维度都做垂域。

**核心动作**：做 3 件事把"通用大模型"变成"行内可用 Agent"。

| 层次 | 动作 | 产出物 | v2 工具 |
| --- | --- | --- | --- |
| **数据层** | 行内私域数据治理：FAQ 知识库、产品手册、合规话术、历史工单、监管处罚案例 | 知识库语料 10 万+ 条；RAG 召回率 ≥90% | Milvus 2.5 + 阿里百炼"数据治理"模块 + 行内数据湖 |
| **模型层** | 底座选型 + 提示词工程 + RAG 检索增强 + LoRA 微调（行内合规话术） | 行内专属提示词模板 200+ 条；RAG 准确率 ≥90% | DeepSeek-V3 + Qwen3-Rerank-8B（重排序） + LangChain 1.0 LCEL |
| **场景层** | 垂域 Agent 编排：客服 Agent / 信贷 Agent / 风控 Agent / 投研 Agent | Agent 编排图 5-10 个；场景上线清单 10+ 场景 | Dify 1.0（多 Agent 编排）/ Coze 扣子（轻量场景） / 行内自研框架 |

**垂域微调 3 步走**（v2 更新版）：

1. **第 1 周**：用行内历史工单 + 客服录音转写（脱敏后）做 RAG，验证"问 100 个真问题，AI 能不能答对 85+"
2. **第 2-3 周**：把 80% 准确率以上的 FAQ 沉淀为"标准答案库"，剩下 20% 走"人在回路"（AI 给建议，人审核后回复）
3. **第 4-6 周**：基于 badcase 做 LoRA 微调（Qwen3-7B 基模 + 行内合规话术 5 万条）+ 知识库补全，准确率目标 ≥95%

**v1 升级点**（v2 新增）：**LoRA 微调场景**——v1 只做"提示词工程 + RAG"，v2 增加"LoRA 微调"作为可选路径（适合准确率要求 ≥95% 的场景，如反欺诈研判、信贷报告生成）。

**拒绝做的事**：
- 不做"通用大模型 demo 给领导看"（领导看了觉得"这不就是 ChatGPT 嘛"）
- 不做"全行一盘棋"（分行/业务部门差异极大，先做 1 个部门 1 个场景跑通）
- 不做"自研底座"（底座用 DeepSeek/Qwen3/智谱/商汤，咨询公司做场景层和数据层）

### 3. Automatic（自动化） —— 把 5 类重复决策从"人判断"变"规则+模型"

**5A 溯源**：Automated（自动化）= "用户端自助分析 + 业务端'数据追人'"（苏春园原话）。银行场景的"数据追人"对应"AI 把决策推送给坐席/客户经理"。

**核心动作**：识别"高频 + 规则清晰 + 量可计算"的决策点，把人从审批流里释放出来。

| 决策类型 | 原本流程 | 自动化后 | v2 工具栈 |
| --- | --- | --- | --- |
| **信用卡反欺诈** | 人工调单 + 风控经理审批（平均 4h） | 模型实时评分（<200ms）+ 灰名单自动拦截 + 人工只复核模型不确定的 5% | Qwen3-7B + LightGBM + 行内规则引擎 + AutoGen 0.4 决策 Agent |
| **小微贷款预审** | 客户经理收材料 + 信贷员审核（平均 2 天） | OCR 自动抽取材料 + 工商/司法/征信接口自动调取 + 模型预审（10 分钟出结论） | 通义 OCR + RPA + 启信宝/天眼查 API + CrewAI 0.80 多 Agent 协作 |
| **保险核保** | 核保员人工查病史 + 出具核保结论（平均 30min/单） | 病历结构化抽取 + 既往症规则匹配 + 模型分级（标准体/次标准体/拒保） | 智谱 GLM-4.6 + 医学知识图谱 + 核保规则引擎 |
| **客服话术合规** | 质检员抽检 2% 录音 | 全量 100% 实时质检 + 违规话术自动标红 + 推送给坐席主管 | 阿里通义语音 + DeepSeek-V3 + 监管口径规则库 + 飞书机器人 |
| **工单分类/优先级** | 客服主管人工分派（平均 5min/单） | 模型自动分类 + 优先级排序 + 智能派单（<1s） | Qwen3-Rerank-8B + 行内工单系统 + 钉钉/飞书 |

**v1 升级点**（v2 新增）：**多 Agent 协作的"反欺诈 + 核保"决策**——v1 是单点模型，v2 用 **CrewAI 0.80** 把"OCR Agent + 知识库 Agent + 决策 Agent + 审计 Agent"组队，完成"材料抽取→规则匹配→决策生成→合规审计"全链路。

**关键纪律**：
- **必须保留"人在回路"**（自动化覆盖率到 80% 就停，剩下 20% 留给"模型不确定"的边界案例）
- **每个自动化决策必须有审计 trail**（金融监管要求所有决策可追溯）
- **灰度上线原则**（先 1 个支行/1 条业务线试点，跑 2 周再扩量）

### 4. Actionable（可执行） —— AI 输出必须能变成"下一步动作"

**5A 溯源**：Actionable（行动化）= "从数据到行动，决策闭环"（苏春园原话）。银行场景的"决策闭环"对应"AI 输出 → 系统触发 → 责任人 → 完成时间 → 反馈回模型"五要素齐全。

**核心动作**：每一条 AI 输出都要绑定"业务动作"+"责任人"+"系统触发"。

| AI 输出类型 | 原本形态 | 变成的可执行动作 | v2 落地形态 |
| --- | --- | --- | --- |
| **客户画像** | "客户 A 偏好理财" | 客户经理下周拜访时直接推送"客户 A 资产 500 万，理财到期 200 万，建议推大额存单" | CRM 系统自动生成待办 + 推送至客户经理飞书 + 自动排程拜访 |
| **信贷风险预警** | "客户 B 风险上升" | 信贷员收到工单"客户 B 经营异常，建议 3 天内现场尽调" | 风控系统自动派单 + 限时 72h 处理 + 超时升级 |
| **客服质检违规** | "坐席 C 存在违规话术" | 主管收到告警"坐席 C 通话 12:34:56 出现承诺收益话术，请立即复盘" | 实时推送主管飞书 + 录音片段定位 + 自动生成复盘工单 |
| **营销活动建议** | "高端客户对境外旅游感兴趣" | 客户经理收到素材包"3 套境外旅游产品文案 + 1 张客户分层名单"，一键群发 | 营销系统预生成素材 + 客户经理审核后触发 + ROI 自动归因 |

**反例（v2 新增 2 条）**：
- ❌ AI 输出"客户有流失风险" → 然后呢？没有"流失谁/怎么挽回/谁负责"，等于没输出
- ❌ AI 输出"建议加大营销投入" → 没有"投多少/投哪个产品/投给谁"，等于没输出
- ❌ AI 输出"某类客户转化率高" → 没有"具体名单 + 触达话术 + 触发时间"，等于没输出（v1 已列）
- 🆕 AI 输出"反欺诈模型命中" → 没有"命中规则/金额阈值/拦截动作/复核人"，等于没输出
- 🆕 AI 输出"贷款预审通过" → 没有"贷款额度/利率/期限/合同生成链接"，等于没输出

### 5. Adaptive（自进化） —— 上线后必须建反馈回路

**5A 溯源**：Adaptive（自适应）↔ Augmented（增强化，原始版）= "AI + 深度学习 + 预测/诊断"（苏春园原话）。"自进化"是"Augmented 增强化"在 2025-2026 Agent 时代的语义延伸——除了 AI 增强人，还强调"AI 增强自己"（通过反馈回路）。

**核心动作**：3 套反馈机制 + 1 个自适应迭代节奏。

| 反馈机制 | 实现方式 | 频率 | v2 工具 |
| --- | --- | --- | --- |
| **坐席评分** | 每次 AI 推荐被坐席采用/忽略/修改，记录到评分库 | 实时 | 行内 IM 集成 + 评分 API + 飞书/钉钉多端 |
| **客户反馈** | 对话结束弹"本次服务是否解决您的问题"，1-5 分 + 文本反馈 | 每次会话 | 客服系统 + NPS 模型 + Dify 1.0 多模态分析 |
| **主管审核** | 主管每周抽检 50 条 AI 对话，标"通过/不通过/需改进" | 每周 | 飞书/钉钉审核工作流 + 数据看板 + CrewAI 0.80 辅助审核 |

**自适应迭代节奏**（v2 升级到每日/每周/每月/每季度）：

- **每日**：自动跑 badcase 聚类（用 DeepSeek-V3 把"低分/违规/不准确"对话聚类成 5-10 个典型问题）→ 推送主管飞书
- **每周**：产品经理 + 业务专家 + 算法工程师 1 小时站会，定本周要修的 top 3 问题
- **每月**：基于月度数据出《模型健康度报告》——准确率、覆盖率、badcase 趋势
- **每季度**：底座模型评估（是否要换基模 Qwen3-235B → Qwen3-Next）+ 知识库大盘点（新增/失效/冲突）

**v1 升级点**（v2 新增）：**底座模型季度评估机制**——2025-2026 大模型迭代极快（DeepSeek-V3 → V3.1、Qwen3 → Qwen3-Next、GPT-4o → GPT-4.5/5），建议每季度评估"是否换基模"。

**关键纪律**：
- **badcase 必须有 owner**（每一条 badcase 必须有明确"谁负责修"+"修到什么时候"）
- **不设"反馈黑洞"**（客户/坐席/主管的反馈必须 7 天内闭环）
- **不允许"模型躺平"**（上线 3 个月内必须看到 badcase 收敛趋势）

## 四、6 周路线图（v2 细化为每日级 check-in）

> **使用方式**：每个工作日 1 个 check-in，总计 30 个工作日（6 周 × 5 天/周）。所有 Day 编号按工作日排列。

### Week 1：数据准备 + 最小可用版本（5 个工作日）

**目标**：1 个场景（推荐：FAQ 自动回复或坐席实时辅助）的最小可用版本，能在行内环境跑起来。

1. **第一刀（最易）**：FAQ 自动回复 / 智能工单分类 —— 投入小、效果立竿见影、风险低
2. **第二刀（中等）**：坐席实时辅助 / 营销素材生成 —— 需要业务部门深度参与，2-3 个月见效
3. **第三刀（高阶）**：信贷报告自动生成 / 智能核保 / 反欺诈辅助 —— 合规要求高，3-6 个月沉淀
4. 🆕 **第四刀（v2 新增）**：投研辅助 / 数字员工 —— 借鉴"森宝"中台 + 钉钉"AI 表格"，搭建"行员 AI 助手"

## 八、可量化 ROI 测算模型（与 v1 保持一致，验证后修订）

| 指标 | 基线（人工） | 目标值（AI 辅助/替代） | 节省/提升 |
| --- | --- | --- | --- |
| 单笔信贷报告 | 4 小时 | 10 分钟 | 效率提升 24x |
| 客服单通电话 | 8 分钟 | 5 分钟 | 效率提升 38% |
| 客服人力替代率 | 100% 人工 | 30-50% AI 自助解决 | 节省 30-50% 人力 |
| 反欺诈识别准确率 | 80% | 95% | 准确率提升 15pp |
| 核保单均处理时间 | 30 分钟 | 8 分钟 | 效率提升 4x |
| 客户 NPS | 40 | 55 | 提升 15pp |

> **测算模型**：按"中等规模城商行"——年客服通话量 500 万通、坐席 200 人、信贷员 50 人。AI 替代率 30% 计算：
> - 客服人力节省：200 × 30% = 60 人 × 15 万/年 = **900 万/年**
> - 信贷员效率提升：50 × 4x 效率 = 节省 37.5 人 × 20 万/年 = **750 万/年**
> - 反欺诈拦截增加：保守估 **500 万/年**
> - **合计年节省 2000+ 万**，项目预算 1000-2000 万，**12 个月内回本**
